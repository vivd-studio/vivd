# Vivd Project State

> Goal: grow Vivd into a reliable hosted website builder first, with Studio, publishing, and plugin interfaces, while keeping a reversible path to a later open-core or self-host split once the product is more stable.

## Current Snapshot

- The control-plane/backend (`packages/backend`) and isolated Studio runtime (`packages/studio`) split is stable, and bucket-first source/preview/publish flow is the baseline.
- Near-term product direction is now platform-first: the hosted `platform` posture is the only intended supported mode for now, while `solo` and public self-hosting are being pushed behind internal or experimental flags instead of treated as co-equal product shapes.
- Fly studio orchestration covers the core hosted lifecycle paths, while Docker/self-host has recent image-reconciliation and runtime-url hardening that now needs repeatable smoke coverage.
- The current architectural reshapes are the Studio preview/runtime split, the OpenCode-aligned chat/runtime refactor, and continued plugin extraction behind generic backend/frontend/CLI contracts.
- Plugin extraction is still useful, and the repo now has an internal `plugins/sdk` boundary for plugin contracts and manifests. Install-profile default policy has now been pulled back out of that SDK surface and into backend host policy, so the next plugin cleanup is simplifying the remaining hosted-platform entitlement/admin flow rather than publishing a profile-aware SDK contract.
- Scratch-to-Studio handoff has been hardened significantly, but the remaining work is proving the attach/build path cleanly across local, CI, and hosted flows.
- Preview and publish artifact preparation currently run through the existing Studio/local build path; the dormant dedicated builder integration has been removed from active backend/studio surfaces until it is revisited with a production-verified plan.

## Active Priorities

1. Execute the platform-first stabilization plan in `plans/platform-first-stabilization-plan.md`: support the hosted `platform` posture as the only intended public mode for now, push `solo` behind experimental flags, remove public self-host emphasis, and simplify plugin/admin policy around the product that is actually being run.
2. Finish the remaining OpenCode-aligned Studio chat/runtime refactor and close the highest-value upstream-parity gaps.
3. Land the Studio preview architecture rework in `plans/studio-preview-architecture-plan.md`, especially the live-preview vs publish-preview split and runtime URL policy.
4. Keep hardening Studio lifecycle across Fly and Docker, especially auth, rehydrate/revert, quiesce, and env/image drift paths.
5. Validate the scratch-to-Studio handoff and the existing preview/publish artifact path end to end before moving more build responsibility off Studio machines.
6. Continue extracting first-party plugins behind generic backend/frontend/CLI boundaries, and keep policy host-owned: defaults and availability should stay in backend instance policy rather than leaking back into plugin contracts or plugin-specific UI copy.
7. Keep only a narrow internal compatibility path for `solo` self-hosting while removing it from the public product and docs posture until there is a stronger reason to support it again.
8. Keep the next control-plane ops tranche queued: reversible project archiving, superadmin project transfer, and post-login tenant redirect, but treat clearly multi-org/shared-control-plane features as platform work rather than as default-core work.
9. Keep the open-core/licensing split as a deferred architectural option rather than the current driver; only continue the subset of boundary cleanup that clearly simplifies the hosted platform codebase today.
