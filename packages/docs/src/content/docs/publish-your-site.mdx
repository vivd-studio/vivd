Publishing is the handoff from draft work inside Studio to the domain your visitors use.

## Before you publish

- Review the latest draft in Studio.
- Confirm that the important pages, sections, and assets are present.
- Check plugin-related setup such as forms or analytics.
- Make sure the target domain is the one you intend to use.

If you are self-hosting, confirm that the install-level host and HTTPS setup are already
correct in [Instance Settings](/instance-settings/) before treating a publish issue as a
project-only problem.
If the target domain itself is unclear, blocked, or still unverified, see
[Domains & Publish Targets](/domains-and-publish-targets/).

## What the publish flow includes

- Previewing the current project state.
- Reviewing the pre-publish checklist.
- Confirming the target domain.
- Pushing the selected version live.

For the domain registration, verification, and allowlist side behind that target-domain
field, read [Domains & Publish Targets](/domains-and-publish-targets/).

## What the checklist is for

The pre-publish checklist is there to catch obvious launch issues before a version goes public. Use it to verify that the site is complete enough for real traffic, not just internally “good enough.”

## Republish flow

After a site is live, you can still return to Studio, refine the project, and publish a newer version when it is ready.

## Expected result

The live domain serves the published version, while Studio remains your editing environment for future changes.

## If something feels off

Use [Troubleshooting](/troubleshooting/) if the live domain does not match what you expected, if the domain setup is unclear, or if a publish looks incomplete.
