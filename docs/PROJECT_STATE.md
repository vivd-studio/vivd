# Vivd Project State (Current)

> Goal: run Vivd as a reliable multi-tenant SaaS with isolated studio machines, object-storage-backed projects, and predictable publish/preview flows.

## Current Focus

- Finish the OpenCode-aligned Studio chat/runtime refactor and close the remaining upstream-parity gaps.
- Harden Studio lifecycle across Fly and Docker, especially auth, rehydrate/revert, and env drift paths.
- Close the remaining scratch-to-Studio initial-generation gaps and validate the dedicated builder path.
- Continue `docs/refactor-and-hardening-plan.md`, starting with auth, transport/state cleanup, and self-host config/source-of-truth cleanup.
- Keep upstream references in `vendor/` useful and keep this file scoped to active work.

## Current Status

- The control-plane/backend (`packages/backend`) and isolated Studio runtime (`packages/studio`) split is stable.
- Bucket-first source, preview, and publish flow is active.
- Fly studio-machine orchestration covers core lifecycle paths; the Docker provider is available but still needs parity hardening.
- Multi-org auth and tenant host scoping are in place across the core control plane.
- The self-host profile split is far enough along that `solo` is the primary self-host target; `platform` remains supported but lower priority.
- The dedicated builder runtime exists behind `VIVD_ARTIFACT_BUILDER_ENABLED` and is still dark-launched.

## Latest Progress

- 2026-03-28: root-caused why some long-broken projects such as `peter-bokel` still stayed unrecovered even on newer Studio images. The newer snapshot repair code was present in the image, but `packages/studio/entrypoint.sh` was hydrating source and OpenCode state in parallel and ran snapshot repair inside the OpenCode branch before source hydration had necessarily restored `.git/opencode`. On the live prod machine that left `/root/.local/share/opencode/snapshot/<project-id>` as an almost-empty placeholder directory (`info/` only, no `HEAD`/`config`/`objects`), so every later reconcile kept skipping real recovery and tracking stayed dead. Snapshot repair now runs only after both source and OpenCode hydration complete, and focused helper regressions now cover both placeholder snapshot dirs and incomplete gitdirs.
- 2026-03-28: completed the full warm-reconcile env fix for Studio machines. The earlier auth split removed `SESSION_TOKEN` drift, but warm reconcile was still rebuilding Fly machines from a base env-only template, which stripped stable project env like `MAIN_BACKEND_URL`, `GITHUB_REPO_PREFIX`, enabled plugin ids, and support email. That left supposedly warmed machines needing a fresh `update -> stopped -> start` cycle on the first real open to restore project env. Studio launch and hard-restart now resolve a shared stable machine env surface, user-scoped organization role is no longer injected into machine env, and Fly warm reconcile now merges the latest stable project env with the machine’s existing stable env fallback before parking it suspended. Targeted backend tests now cover canonical callback URL wiring, sorted plugin env, and warm reconcile preserving project env for the first wake.
- 2026-03-28: identified and fixed the next Fly warm-wake drift source after the Studio auth split. `SESSION_TOKEN` was no longer present on parked Studio machines, but default-org projects such as `bettinis-vintage` could still get forced through `image,env` warm reconcile because `MAIN_BACKEND_URL` was being derived from the incoming request host, so opening the same Studio through `vivd.studio` vs `default.vivd.studio` changed machine env and triggered a fresh boot path. Remote Studio machines now prefer the configured canonical backend origin (`BACKEND_URL` / `DOMAIN` / `BETTER_AUTH_URL`) for `MAIN_BACKEND_URL`, and enabled plugin ids are now sorted before being written into env so plugin row order cannot cause spurious env drift either. Targeted backend callback-url and Studio launch tests now cover both stabilizations.
- 2026-03-28: hardened the Studio OpenCode chat recovery path for stale visible tabs. When a tab becomes visible after inactivity and the event stream looks stale, the client now restarts the subscription instead of only queueing a snapshot refresh, which is closer to upstream OpenCode web's visibility-triggered reconnect behavior. The selected-session controller also now runs a visible-session stale watchdog: if a session stays locally `Working...` for too long without any new stream activity, it refetches both bootstrap/session state so replies that already finished on the server do not stay wedged behind stale local sync until a manual browser refresh.
- 2026-03-28: corrected the backend image smoke's migration assertion. The smoke was proving startup by querying `__drizzle_migrations`, but that tied the release gate to Drizzle's internal journal table rather than the actual runtime contract and failed on fresh smoke databases where the journal table was absent even though backend startup had already reached health. `scripts/backend-image-smoke.mjs` now verifies that startup initialized a representative set of core application tables instead, which is a stronger and less implementation-coupled proof that the image can boot against a fresh database.
- 2026-03-28: fixed the backend release-image smoke crash caused by eager OpenRouter client initialization. `packages/backend/src/services/integrations/OpenRouterService.ts` had been constructing the `openai` client at module load, which made backend startup hard-fail with `Missing credentials` whenever `OPENROUTER_API_KEY` was absent, even in environments that did not need generation features just to boot. The OpenRouter client is now initialized lazily, and the service only throws when an actual OpenRouter-backed operation is invoked without credentials, so backend startup/smoke no longer depends on unused AI secrets being present.
- 2026-03-28: simplified the main tag-release publish policy in `.github/workflows/publish.yml`. Real `v*` tag pushes no longer use selective image rebuilds from `detect-changes`; they now always rebuild, smoke, and publish the full amd64 image set so a failed previous tag or an overly narrow diff cannot leave part of a release image matrix stale. The selective diff path is still kept for manual workflow dispatches, where a narrower rebuild can still be useful.
- 2026-03-28: added a guarded local tag publish helper in `scripts/publish.sh` and exposed it as `npm run publish:tag -- <version>`. The default `release` preflight now sticks to the current release-critical gate: root typecheck plus the targeted Studio auth and backend Studio runtime/Fly regressions already used in CI, so local tag publishing stays usable while broader frontend lint and some unrelated backend unit tests still have baseline failures; `--check-mode ci-local` remains available for the broader local lint/typecheck/unit/full-Studio gate.

The test now closes over an explicit headers record instead of relying on method `this`, so Studio server typecheck stays green in validate.
